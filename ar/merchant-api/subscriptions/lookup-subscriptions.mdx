# البحث عن اشتراكات

<div className="flex items-center gap-2 mb-4">
  <span className="px-2 py-1 bg-green-100 text-green-800 rounded text-sm font-mono font-bold">GET</span>
  <code>/subscriptions/lookup</code>
</div>

يُعيد هذا المسار الاشتراكات التي يمتلكها مستخدم نهائي معيّن في متجرك، مع تصفيتها افتراضيًا لتشمل فقط الاشتراكات التي تمنح الوصول حاليًا (الحالة `active` أو `trialing`).

هذا هو المسار الساخن (hot-path) لـ **جدران الدفع (paywalls) في تطبيقات SaaS** — إذ يسأل التكامل النموذجي "هل لدى هذا المستخدم اشتراك نشط؟" عند كل طلب محميّ. تتضمن الاستجابة حقلًا منطقيًا ملائمًا باسم `has_active` يتيح لك التحكّم في الوصول بفحص حقل واحد فقط.

## الترويسات (Headers)

| Header | القيمة | مطلوب |
|--------|-------|----------|
| `Authorization` | `Bearer YOUR_API_TOKEN` | نعم |
| `Accept` | `application/json` | مستحسن |

## حدّ المعدّل (Rate Limit)

هذا المسار محدَّد بـ **120 طلبًا في الدقيقة** لكل رمز API — وهو أعلى من بقية مسارات الاشتراكات لأنه مُصمَّم لاستدعائه عند كل طلب خلف جدار الدفع. إن كنت تتوقّع حجمًا أعلى، فاحتفظ بنتائج الاستعلام مؤقتًا (cache) من جانبك لفترة قصيرة (مثل 60 ثانية لكل مستخدم).

## معاملات الاستعلام (Query Parameters)

يجب تمرير **معرّف واحد على الأقل** من المعرّفات التالية للمستخدم:

- `external_customer_id`، **أو**
- `email`، **أو**
- `phone` (مع `country_code`).

إذا لم يُمرَّر أيّ من هذه المعرّفات، فسيُرفض الطلب بـ `422`.

| Parameter | النوع | مطلوب | الوصف |
|-----------|------|----------|-------------|
| `external_customer_id` | string (حتى 191 حرفًا) | مشروط | معرّف المستخدم الخاص بالتاجر، المخزَّن على الاشتراك عند إنشاء جلسة الدفع. هذا أسرع وأوثق وسيلة للبحث لأنه مفهرَس ومعزول لكل متجر. |
| `email` | email (حتى 191 حرفًا) | مشروط | البريد الإلكتروني للعميل. تُجرى مطابقة دقيقة مع سجل العميل. |
| `country_code` | string (حتى 5 أحرف) | مشروط | مفتاح الاتصال الدولي بدون `+` (مثال: `966`). تُزال العلامة `+` الأولى تلقائيًا. يجب إقرانه مع `phone` ليكون ذا فائدة. |
| `phone` | string (حتى 15 رقمًا) | مشروط | رقم جوّال العميل (الجزء المحلي فقط، بدون مفتاح الدولة). |
| `product_id` | integer | لا | يحصر النتائج بمنتج واحد. مفيد عندما تبيع عدة منتجات اشتراك وتريد التحقق من حقّ الوصول لأحدها فقط. |
| `include_inactive` | boolean | لا | القيمة الافتراضية `false`. عند `false`، تُعاد الاشتراكات ذات الحالة `active` و`trialing` فقط (عرض "الممنوح حق الوصول حاليًا"). عيّنها إلى `true` لإظهار الاشتراكات ذات الحالات `paused` و`past_due` و`canceled` و`expired` أيضًا، مثلًا عند بناء شاشة سجلّ الفوترة. |

<Note>
البحث يقتصر دائمًا على نطاق متجرك — لن ترى أبدًا اشتراكات تعود لتاجر آخر، حتى إن كان البريد الإلكتروني للعميل مستخدَمًا في متاجر متعددة.
</Note>

<Info>
عند تمرير `email` و`phone` معًا، تُطبَّق بينهما علاقة "AND" داخل نفس الاستعلام المرتبط بالعميل (whereHas)، أي أن سجل العميل يجب أن يطابق الاثنين. كما أن النتائج محدودة بحدٍّ أقصى **50 اشتراكًا** في كل استدعاء.
</Info>

## أمثلة على الطلبات

### فحص جدار الدفع بالمعرّف الخارجي (الأكثر شيوعًا)

<CodeGroup>
```bash cURL
curl -X GET "https://merchant-api.rmz.gg/shawarma/subscriptions/lookup?external_customer_id=usr_abc123" \
  -H "Authorization: Bearer YOUR_API_TOKEN"
```

```javascript JavaScript
const params = new URLSearchParams({
  external_customer_id: "usr_abc123"
});

const response = await fetch(
  `https://merchant-api.rmz.gg/shawarma/subscriptions/lookup?${params}`,
  { headers: { "Authorization": "Bearer YOUR_API_TOKEN" } }
);
const { data } = await response.json();

if (data.has_active) {
  // grant access
}
```

```python Python
import requests

response = requests.get(
    "https://merchant-api.rmz.gg/shawarma/subscriptions/lookup",
    headers={"Authorization": "Bearer YOUR_API_TOKEN"},
    params={"external_customer_id": "usr_abc123"},
)
data = response.json()["data"]

if data["has_active"]:
    grant_access()
```

```php PHP
$params = http_build_query(["external_customer_id" => "usr_abc123"]);
$ch = curl_init("https://merchant-api.rmz.gg/shawarma/subscriptions/lookup?{$params}");
curl_setopt($ch, CURLOPT_HTTPHEADER, ["Authorization: Bearer YOUR_API_TOKEN"]);
curl_setopt($ch, CURLOPT_RETURNTRANSFER, true);
$response = json_decode(curl_exec($ch), true);

if ($response["data"]["has_active"]) {
    // grant access
}
```
</CodeGroup>

## حقول الاستجابة

| Field | النوع | الوصف |
|-------|------|-------------|
| `count` | integer | عدد الاشتراكات المُعادة. يُحدّد المسار النتائج بحدٍّ أقصى **50** اشتراكًا في كل استدعاء. |
| `has_active` | boolean | يكون `true` إذا كانت إحدى الاشتراكات المُعادة على الأقل بالحالة `active` أو `trialing`. استخدمه كبوابة جدار الدفع لديك. |
| `subscriptions` | array | مصفوفة من كائنات الاشتراك. لكل كائن نفس شكل [عرض اشتراك](./get-subscription)، **دون** حقول `payment_method` و`scheduled_variant` و`customer` (تُحمَّل مسبقًا هنا علاقات `product` و`variant` فقط). |

### النتيجة الفارغة

عندما لا يُطابق أي اشتراك الشروط، يُعيد المسار `200 OK` مع مصفوفة فارغة — ولا يُعاد `404`:

```json
{
  "message": null,
  "data": {
    "count": 0,
    "has_active": false,
    "subscriptions": []
  },
  "api": "rmz.shawarma",
  "timestamp": 1699999999
}
```

## نمط تكامل شائع

وسيطة (middleware) بسيطة لجدار الدفع في تطبيق SaaS:

```javascript
async function hasActiveSubscription(userId) {
  const res = await fetch(
    `https://merchant-api.rmz.gg/shawarma/subscriptions/lookup?external_customer_id=${encodeURIComponent(userId)}&product_id=102`,
    { headers: { Authorization: `Bearer ${process.env.RMZ_API_TOKEN}` } }
  );

  if (!res.ok) {
    // Fail closed or fail open depending on your risk tolerance.
    // Most paywalls fail closed on 5xx and fail open on network errors.
    return false;
  }

  const { data } = await res.json();
  return data.has_active;
}
```

<Tip>
مرِّر `external_customer_id` عند إنشاء جلسة الدفع (راجع [إنشاء جلسة دفع](./create-checkout-session)). يُخزَّن هذا المعرّف على الاشتراك نفسه، مما يجعل البحث قراءة مفهرَسة مباشرة بدلًا من ربط بجدول `customers`.
</Tip>

## استجابات الأخطاء

| Code | الوصف |
|------|-------------|
| `401` | غير مصرّح — رمز API غير صالح أو مفقود. |
| `403` | فوترة الاشتراكات غير مفعَّلة للمتجر (يتطلّب RMZ Plus). |
| `422` | خطأ في التحقق. يُعاد عند عدم تمرير أيّ معرّف، أو عند تجاوز معامل الحد الأقصى لطوله أو فشل فحص نوعه (مثلًا أن تكون قيمة `email` بريدًا إلكترونيًا غير صالح). |
| `429` | تجاوز حدّ المعدّل (120 طلبًا في الدقيقة لكل رمز). |

### مثال `422` — لم يُمرَّر أيّ معرّف

```json
{
  "message": "At least one of external_customer_id, email, or phone is required",
  "data": null,
  "api": "rmz.shawarma",
  "timestamp": 1699999999
}
```

### مثال `422` — معامل غير صالح

```json
{
  "message": "The given data was invalid.",
  "errors": {
    "email": ["The email must be a valid email address."]
  }
}
```
